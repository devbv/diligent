## Structural Integrity

### 프로젝트 구조가 skill 내부에 하드코딩됨
**Impact**: Compounds over time

`packages/core/src/`, `packages/cli/src/`, `research/layers/`, `research/llm-tools/` 등 실제 파일 경로들이 skill 본문에 박혀 있습니다. 프로젝트 구조가 바뀌면 "What to Read" 섹션이 잘못된 경로를 안내하는 dead code가 됩니다. 코드 컴파일 오류처럼 명시적으로 실패하지 않아 아무도 눈치채지 못합니다.

Fix: ARCHITECTURE.md를 먼저 읽고 현재 구조를 파악하도록 지시하는 동적 접근으로 변경.

### 숫자가 굳어 있음
**Impact**: Cosmetic → Compounds

- `"78+ numbered design decisions"` — time-bomb
- `"11 layers (L0-L10)"` — 레이어 추가 시 틀림
- `"6 phases (P0-P5)"` — 계획 변경 시 틀림

Fix: 살아있는 문서를 참조하도록 대체.

---

## Velocity Trajectory

### Context Gathering이 unsized
**Trend**: Degrading as project grows

"Build a complete picture before forming judgments" — 우선순위 없이 전부 읽으라는 지시. 프로젝트가 커질수록 비효율 증가.

Fix: `STATUS.md → git log → next phase spec → code (targeted)` 순서로 계층화.

### Rework ratio 계산 기준 미정의
**Trend**: Flat

"Calculate this explicitly"라고 했지만 무엇이 rework인지 기준 없음. 매 실행마다 다른 결과.

Fix: `rework = (delete + rename + reorganize commits) / total commits` 명시.

## Forward Compatibility

### Output에 완료 기준 없음
**Horizon**: Breaks immediately in practice

Finding 개수에 상한 없음. 10페이지짜리 리포트는 아무도 액션하지 않음.

Fix: 섹션당 최대 2-3개, Priority Actions 최대 3개로 cap 설정.

### Axis 3 (Forward Compatibility)는 evidence 원칙과 충돌
**Horizon**: Every run

"Evidence, always" 선언했지만 Forward Compatibility는 본질적으로 미래 예측. `[HYPOTHESIS]` 태그로 speculative 항목을 명시적으로 구분.

### 반복 실행 시 이전 결과와 비교 불가
**Horizon**: Phase 3+

Sustainable developability는 추세인데 스냅샷만 찍음. → 이 review/ 디렉터리 구조로 해결.
